# Technical Design: Fix Loop Tiered Re-entry

## Context

The `specshift apply` Fix Loop currently provides no classification of correction severity. When a PR reviewer points out an approach change (wrong files modified, wrong abstraction), the agent lacks explicit guidance to escalate beyond patching. The result is artifact drift: design.md, tasks.md, and review.md remain consistent with the original wrong approach even after multiple fix commits.

Three files require changes:
1. `docs/specs/human-approval-gate.md` — Fix Loop requirement (adds tiered classification + artifact staleness rule)
2. `src/templates/workflow.md` + `.specshift/WORKFLOW.md` — apply instruction (adds tier classification step + artifact staleness rule)
3. `src/templates/changes/tasks.md` + `.specshift/templates/changes/tasks.md` — Fix Loop step 3.4 (adds tier vocabulary to the checklist item)

After spec/template edits, `bash scripts/compile-skills.sh` must be run to regenerate the AOT-compiled action file at `.claude/skills/specshift/actions/apply.md`.

## Architecture & Components

| File | Change |
|------|--------|
| `docs/specs/human-approval-gate.md` | Update Fix Loop requirement: add Tier 1/2/3 definitions, detection signals, artifact staleness rule; add three new Gherkin scenarios |
| `src/templates/workflow.md` | Replace apply instruction's "Fix loop" + "Artifact freshness" lines with tiered classification instruction |
| `.specshift/WORKFLOW.md` | Sync the same apply instruction change from src |
| `src/templates/changes/tasks.md` | Update step 3.4 description with tier vocabulary |
| `.specshift/templates/changes/tasks.md` | Sync the same step 3.4 change from src |
| `.claude/skills/specshift/actions/apply.md` | Regenerated by `bash scripts/compile-skills.sh` (AOT compilation) |

No new files. No new specs. No changes to other specs or pipeline stages.

## Goals & Success Metrics

- **G1**: `human-approval-gate.md` Fix Loop requirement defines Tweak / Design Pivot / Scope Change with descriptions and detection signals — PASS/FAIL
- **G2**: `human-approval-gate.md` includes at least 2 new Gherkin scenarios for tiered re-entry — PASS/FAIL
- **G3**: `src/templates/workflow.md` apply instruction contains "Tweak", "Design Pivot", and "Scope Change" tier vocabulary and artifact staleness rule — PASS/FAIL
- **G4**: `src/templates/changes/tasks.md` step 3.4 references tier classification — PASS/FAIL
- **G5**: `.specshift/WORKFLOW.md` and `.specshift/templates/changes/tasks.md` are in sync with their `src/` counterparts — PASS/FAIL
- **G6**: `bash scripts/compile-skills.sh` runs without errors and regenerates `.claude/skills/specshift/actions/apply.md` — PASS/FAIL

## Non-Goals

- No new workflow action for "re-entry"
- No changes to the `specshift propose` pipeline or artifact format
- No changes to how specs are formatted (spec-format.md)
- No changes to the QA Loop steps 3.1–3.3, 3.5, 3.6
- No changes to the review template or preflight template

## Decisions

| Decision | Rationale | Alternatives |
|----------|-----------|--------------|
| Update apply instruction rather than adding a new action | The fix loop is part of apply; a new action would fragment the UX and break the existing checklist flow | New `specshift reenter` action — over-engineered |
| Three tiers (not two) | Tweak vs. re-entry is too coarse — PR #12 involved a Design Pivot (not a full Scope Change). Three tiers allow precision without complexity | Two tiers (fix/re-enter), four tiers — too coarse or too fine |
| Detection signals as observable facts | Agents can check signals mechanically; pure subjective judgment failed in PR #12 | Judgment-only classification — shown to fail |
| Update tasks template step 3.4 description | The step is already in the checklist; enhancing its description makes the tier vocabulary visible at implementation time | Leave template unchanged — misses the direct user-facing instruction location |

## Risks & Trade-offs

- **Tier boundary ambiguity** → Mitigation: artifact staleness rule errs toward higher tier; "ambiguous → Design Pivot" edge case documented in spec
- **Template drift** (src vs .specshift not in sync) → Mitigation: explicit sync tasks in tasks.md; AOT compilation as final validation

## Migration Plan

No migration needed. The changes are to instruction text and spec requirements. Existing changes in progress are not affected; tiered re-entry applies to future fix loop invocations.
