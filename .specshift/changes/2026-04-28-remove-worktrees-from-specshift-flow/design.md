# Technical Design: Remove Worktrees from SpecShift Workflow

## Context

SpecShift currently owns a complete worktree lifecycle (creation, lazy stale cleanup, post-merge cleanup, change-context fallback) gated on `worktree.enabled` in WORKFLOW.md. The feature spans:

- 4 config keys (`worktree.enabled`, `worktree.path_pattern`, `worktree.auto_cleanup`, `worktree.stale_days`)
- All 4 workflow actions (propose, apply, finalize, review)
- The router's Change Context Detection (worktree-name fallback)
- 5 specs and 2 capability docs
- The compiled release tree at `skills/specshift/`

The hosts that run SpecShift (Claude Code, Codex CLI) and plain `git` already provide native worktree affordances, so SpecShift's lifecycle adds surface area without unique value. This change removes the lifecycle while preserving the file-based change workspace under `.specshift/changes/`.

## Architecture & Components

The propose-phase artifact churn lands in two places:

1. **Specs** (`docs/specs/`) — five files trimmed in the prior step (the specs commit on this branch).
2. **Plugin source** (`src/`) — to be edited during the apply phase. The same edits then propagate into the compiled release tree at `skills/specshift/` when finalize runs `bash scripts/compile-skills.sh`.

### Affected layers

| Layer | File(s) | Change |
|-------|---------|--------|
| Skill source | `src/skills/specshift/SKILL.md` | Drop `worktree` config key; drop the worktree-convention fallback (step 3) from Change Context Detection; renumber subsequent steps. |
| Action source | `src/actions/propose.md` | Drop "Create Worktree-Based Workspace", "Lazy Worktree Cleanup at Change Creation", and worktree language from "Create Change Workspace" / "Change Context Detection". |
| Action source | `src/actions/finalize.md` | Drop post-merge worktree cleanup. |
| Action source | `src/actions/review.md` | Drop the worktree cleanup step in the merge sequence; keep branch deletion. |
| Action source | `src/actions/init.md` | Drop worktree opt-in, GitHub merge-strategy config tied to worktree opt-in, and worktree-related env checks (git 2.5+, `/.claude/` in `.gitignore`). |
| Workflow template | `src/templates/workflow.md` | Remove the commented-out `worktree:` block; bump `template-version` per CONSTITUTION's "Template-version discipline". |
| Project workflow | `.specshift/WORKFLOW.md` | Remove the live `worktree:` block (`enabled: true`, `auto_cleanup: true`). |
| Project doc | `AGENTS.md` | Remove the `worktree.enabled: true` reference from the "File Ownership" entry for `.specshift/`. |
| Proposal template | `.specshift/templates/changes/proposal.md` | Drop `worktree` from the proposal-tracking frontmatter documentation. |
| Compiled release | `skills/specshift/` | Regenerated by `bash scripts/compile-skills.sh` during finalize. Not edited directly. |

### Change Context Detection — before vs after

```
Before (3 tiers):
  1. proposal frontmatter `branch:` lookup
  2. worktree convention (git rev-parse --git-dir contains /worktrees/, derive name from branch)
  3. directory listing prompt

After (2 tiers):
  1. proposal frontmatter `branch:` lookup
  2. directory listing prompt
```

Tier 1 already covers every change created since worktree adoption, so dropping tier 2 has no functional impact for in-flight or future changes; only legacy proposals without `branch:` frontmatter would have used tier 2, and they fall through to tier 3 cleanly.

## Goals & Success Metrics

* `git grep -i worktree -- src/ .specshift/WORKFLOW.md AGENTS.md .specshift/CONSTITUTION.md .specshift/templates/changes/proposal.md README.md` returns **0 hits** after apply.
* `git grep -i worktree -- docs/specs/ docs/capabilities/ skills/specshift/` returns **0 hits** after finalize re-runs `bash scripts/compile-skills.sh`. Migration documentation about the now-removed `worktree:` proposal-frontmatter field lives in `CHANGELOG.md` only — specs describe the current contract via a generic "skills ignore unknown frontmatter fields" rule rather than naming the removed field.
* `bash scripts/compile-skills.sh` exits 0 (template-version bump on `src/templates/workflow.md` is detected; no compile-time validation failures).
* All 5 affected specs retain ≥3 requirements each (CONSTITUTION's "consolidation floor").
* Existing `proposal.md` files in `.specshift/changes/*/` keep their legacy `worktree:` frontmatter unchanged (no migration touch); they still load via the new generic ignore-unknown-fields rule.

## Non-Goals

* **Host-level worktree usage**: not restricted. Claude Code's `Agent` tool isolation, Codex's worktree flow, and `git worktree` remain available to the user; SpecShift simply does not orchestrate them.
* **Migration tooling**: in-flight proposals that already carry `worktree: <path>` frontmatter are not rewritten. They remain valid as legacy artifacts.
* **Cleanup of existing worktrees on disk**: SpecShift no longer auto-cleans; users may run `git worktree remove <path>` manually if needed.
* **Historical `.specshift/changes/2026-03-30-worktree-based-change-lifecycle/` and related dirs**: untouched as historical record.
* **Change-workspace concept**: stays file-based under `.specshift/changes/`. Not reworked.
* **Rebase-merge config**: the GitHub Merge Strategy Configuration requirement was tied to worktree opt-in and is removed alongside it. SpecShift still uses squash-merge in the review action.

## Decisions

| Decision | Rationale | Alternatives |
|----------|-----------|--------------|
| Remove worktree handling end-to-end (lifecycle + config + spec material) instead of gutting the implementation while keeping config keys. | Dead config keys would silently no-op for users who set `worktree.enabled: true`, which is worse than removing them outright. The breaking change is documented in the changelog. | Keep config keys, no-op the implementation — rejected (worse UX). |
| Treat existing proposals' `worktree:` frontmatter as legacy/read-only rather than rewriting them. | Migration tooling is out of scope per the issue and would mutate committed history without benefit. The router's primary path (`branch:` lookup) is unaffected by the field's presence. | Strip the field on read or in a one-shot rewrite — rejected (history mutation, no value). |
| Drop the worktree-convention fallback from Change Context Detection and rely on `branch:` frontmatter. | Every proposal created since worktree adoption carries `branch:`. Legacy proposals without it fall through to the directory-listing prompt cleanly. | Keep the fallback — rejected, would require keeping `git rev-parse --git-dir /worktrees/` detection logic in the router. |
| Remove the GitHub Merge Strategy Configuration requirement (rebase-merge). | It was triggered exclusively by worktree opt-in during init, and the review action merges via squash anyway, so the rebase-merge config has no remaining caller. | Decouple it from worktree opt-in and keep it as standalone optional config — rejected, the requirement does not align with how the review action actually merges. |
| Bump `src/templates/workflow.md` `template-version` even though only commented content is removed. | CONSTITUTION's "Template-version discipline" requires a bump on any modification to `src/templates/`, and the compile script enforces it. | Skip the bump because the change is "comment only" — rejected, would break the compile gate. |
| Re-run `bash scripts/compile-skills.sh` during finalize to regenerate `skills/specshift/`. | Standard finalize behavior; the compiled tree must reflect spec/template cleanup. | Regenerate manually before finalize — rejected, redundant. |

## Risks & Trade-offs

* **Edits span 12+ files.** A single mismatched edit can leave dangling worktree language in the compiled tree → Mitigation: post-apply `git grep -i worktree` against `src/`, `.specshift/`, `AGENTS.md`, and (after recompile) `skills/specshift/` is part of the audit gate.
* **Breaking config change.** Consumers with `worktree.enabled: true` in their WORKFLOW.md silently lose the feature → Mitigation: explicit changelog entry under "Breaking" calling out that the config key is no longer read.
* **Spec linkage in compiled actions.** `src/actions/*.md` files link into `docs/specs/*.md`; out-of-date spec snippets baked into the compiled tree would be a regression → Mitigation: finalize re-runs the compile script which propagates the cleaned spec content.
* **Template-version compile gate.** Forgetting to bump `src/templates/workflow.md` template-version would fail compilation → Mitigation: explicit task in tasks.md and pre-finalize sanity check.

## Migration Plan

1. **For SpecShift consumers** (post-merge): on next `specshift init`, the `worktree:` block is no longer copied into new projects. Existing projects keep their local `WORKFLOW.md` unchanged until the user re-runs init; the orphaned `worktree:` block is silently ignored. Consumers who actively used worktree mode should remove their on-disk worktrees manually (`git worktree remove <path>`).
2. **For this project**: the live `worktree:` block in `.specshift/WORKFLOW.md` is removed in the apply step. No on-disk worktrees exist on this branch (working in main worktree).
3. **Rollback**: revert the merged squash commit. The proposal artifacts on this branch (`proposal.md`, `research.md`, `design.md`, `tasks.md`, `preflight.md`) stay as historical record either way.
