# Technical Design: Post PR Summary Comment

## Context

The review action needs a new phase between CI gate and merge confirmation that posts a summary comment on the PR. This is a spec + template change — no executable code. The change also converts the review action's numbered steps to descriptive phase labels to eliminate fragile renumbering.

Files affected:
- `docs/specs/review-lifecycle.md` — new requirement (already done in specs stage)
- `src/actions/review.md` — new requirement link
- `src/templates/workflow.md` — template-version bump + step format conversion + new phase
- `.specshift/WORKFLOW.md` — mirror template changes with local customizations preserved
- `.claude/skills/specshift/actions/review.md` — regenerated by compilation
- `.claude/skills/specshift/templates/workflow.md` — regenerated by compilation

## Architecture & Components

### Spec layer (`docs/specs/review-lifecycle.md`)
New requirement "Pre-Merge Summary Comment" inserted between "Review Cycle Safety Limit" and "Merge Execution with Mandatory Confirmation". Includes 4 scenarios covering normal posting, zero-count, failure graceful degradation, and re-entrant idempotency.

### Action link layer (`src/actions/review.md`)
New link `- [Pre-Merge Summary Comment](...)` between Safety Limit and Merge Execution links. The compile script extracts the requirement block via this link.

### Workflow instruction layer (`src/templates/workflow.md` + `.specshift/WORKFLOW.md`)
Convert numbered steps 1-8 to descriptive phase labels:
- `1. If PR is draft:` → `- **Draft transition:** If PR is draft,`
- `6. If reviewer posts new comments: process them (return to step 4).` → `- **Cycle limit:** If reviewer posts new comments, return to Comment processing.`
- Old step 7 splits into **CI gate** and **Pre-merge summary**
- Old step 8 becomes **Merge execution**

New phase added:
```
- **Pre-merge summary:** If CI is passing, post a summary comment on the PR...
```

### Compilation layer
`bash scripts/compile-skills.sh` extracts the new requirement block and copies the updated workflow template.

## Goals & Success Metrics

* Compilation succeeds with 9 requirement links extracted (up from 8)
* Compiled action contains "Pre-Merge Summary Comment" requirement block
* Compiled workflow template contains "Pre-merge summary" phase label
* Spec passes format validation: normative text before user story, `####` scenario headings, visible assumption text with HTML comment tag

## Non-Goals

- Runtime implementation of comment-posting logic (specs guide agent at runtime)
- Configuration surface for enabling/disabling summary (always-on)
- Changes to other actions (finalize, apply, etc.)
- New ADR for this change (single-requirement addition, no architectural decision)

## Decisions

| Decision | Rationale | Alternatives |
|----------|-----------|--------------|
| Graceful failure on post error | Follows Review Request Dispatch pattern — log warning, continue. Summary is valuable but not critical-path. | Hard failure blocking merge |
| Idempotent via `<!-- specshift:review-summary -->` marker | Standard GitHub bot pattern; avoids duplicate comments across re-entrant sessions | Post new comment each time; delete and repost |
| Cumulative counts from GitHub state | Consistent with re-entrancy requirement — no session-local storage | Session-scoped counts only |
| Phase labels instead of numbered steps | Eliminates fragile renumbering; clearer back-references; consistent with other actions | Keep numbered steps and renumber |
| Zero-count summary still posted | Confirms action assessed the PR even with no reviewer comments | Skip summary when nothing to report |

## Risks & Trade-offs

- **Phase label migration**: Existing sessions referencing "step 4" or "step 7" will see different wording after this change. Risk is minimal — the review action is re-entrant and reads current state, not step history. Mitigation: the phase labels are more descriptive than numbers, so no behavior change.
- **Marker detection reliability**: The HTML comment `<!-- specshift:review-summary -->` must survive GitHub's comment rendering. Risk is very low — GitHub preserves HTML comments in issue/PR comment bodies.
