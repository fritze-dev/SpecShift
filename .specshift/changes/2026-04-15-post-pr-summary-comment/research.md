# Research: Post PR Summary Comment

## 1. Current State

The `specshift review` action is a re-entrant PR review-to-merge state machine defined across multiple files:

- **Spec**: `docs/specs/review-lifecycle.md` — 6 requirements (PR State Assessment, Draft-to-Ready, Review Request Dispatch, Review Comment Processing, Review Cycle Safety Limit, Merge Execution with Mandatory Confirmation)
- **Source action**: `src/actions/review.md` — 8 requirement links compiled into the release directory
- **Workflow template**: `src/templates/workflow.md` (lines 86-101) — review action instruction with numbered steps 1-8
- **Project instance**: `.specshift/WORKFLOW.md` (lines 85-100) — mirrors template with local customizations (Copilot config, worktree cleanup detail)
- **Compiled action**: `.claude/skills/specshift/actions/review.md` — generated by `scripts/compile-skills.sh`, extracts requirement blocks from specs

Currently, the review action processes comments, runs self-checks, and asks for merge confirmation — but only reports results in the chat session. No PR comment is posted, so there's no audit trail visible on GitHub.

### Key patterns observed

- **Tool-agnostic language** (CONSTITUTION.md line 52): Specs say "using available GitHub tooling (gh CLI, MCP tools, or API)" — never hardcode tool names.
- **Graceful degradation**: Review Request Dispatch logs a warning and continues if request fails — same pattern should apply to summary posting.
- **Re-entrancy**: All state derived from GitHub and local artifacts; no session-local state stored.
- **Numbered steps fragility**: Review action uses numbered steps 1-8 with back-reference at step 6 ("return to step 4"). Other actions (propose, init, apply) use prose without numbers.

### Existing PR comment patterns

No existing action posts PR comments. The CONSTITUTION.md standard tasks mention "Reply to and resolve all PR review comments" but this is a manual checklist item, not automation.

## 3. Approaches

| Approach | Pro | Contra |
|----------|-----|--------|
| New requirement in existing `review-lifecycle.md` spec + updated workflow instructions | Follows existing pattern; single spec file covers all review behavior; minimal file churn | Spec file grows longer (~40 new lines) |
| Separate spec for "PR audit trail" capability | Clean separation of concerns | Over-engineering for a single requirement; violates consolidation rules (<3 requirements) |

**Recommendation**: Approach 1 — add to existing spec. The Pre-Merge Summary Comment is a single requirement within the review lifecycle domain.

## 4. Risks & Constraints

- **Comment marker idempotency**: The `<!-- specshift:review-summary -->` marker must be reliably detectable across re-entrant sessions to avoid duplicate comments. Risk is low — HTML comments are stable in GitHub's API.
- **Counting accuracy**: Cumulative counts (threads resolved, fixes applied) must be derived from GitHub state, not session memory. This is consistent with existing re-entrancy design.
- **Template-version bump**: Modifying `src/templates/workflow.md` requires incrementing `template-version` in frontmatter (enforced by compilation).
- **Step numbering refactoring**: Converting numbered steps to phase labels is a broader change but is additive, not breaking — existing behavior is preserved.

| 1 | Add to existing `review-lifecycle.md` spec | Consolidation check: <3 standalone requirements, same domain | Separate spec file |
