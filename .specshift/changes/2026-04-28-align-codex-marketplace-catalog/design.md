# Technical Design: Align Codex Marketplace Catalog Documentation

## Context

The Codex marketplace catalog file `.agents/plugins/marketplace.json` was committed in `71c000fc` and verified functional by the user — `codex plugin marketplace add fritze-dev/SpecShift` resolves the plugin via the catalog's Git-URL source. (The official Codex CLI plugin docs at `developers.openai.com/codex/plugins/build` document the `owner/repo` shorthand without `github:` prefix, and install/enable happens via the in-session `/plugins` directory rather than a separate `plugin install <name>` CLI command.) However, the authoritative documents that describe shipped behavior still claim the file is not shipped. This change is purely textual reconciliation: align five documents (two specs, AGENTS.md, CONSTITUTION.md, README.md) plus a small ADR amendment with the committed reality. No runtime behavior changes; no build script extension.

The constraint set is conservative by design (per user direction):
- The committed catalog file's contents stay as-is — we describe what is, we don't redesign it.
- `scripts/compile-skills.sh` and `.github/workflows/release.yml` get **no** code changes. The script's misleading "four root files" header comments are tolerated until a separate cleanup issue.
- PR #52, PR #53, Issue #51 stay open — research-only inputs.

## Decisions

| Decision | Rationale | Alternatives |
|----------|-----------|--------------|
| Codex catalog uses `source: { source: "url", url: "<repo>.git" }` | User-validated functional. Codex re-clones the URL during install and resolves the existing `.codex-plugin/plugin.json` and `./skills/specshift/` at the repo root — no sub-payload replication needed. Symmetric to Claude's external-plugin pattern. | `local` path (`../../.codex-plugin` PR #52 / `./plugins/specshift` PR #53). PR #53's path requires a generated sub-payload — extra complexity for no benefit when the URL form already works. |
| Catalog policy `AVAILABLE` / `ON_INSTALL` | Ratified by PR #53's audit (it corrected PR #52's `user-required`/`none`). Matches the committed file. | `user-required` / `none` (PR #52 — rejected, doesn't match Codex install semantics). |
| Catalog has no `version` field; not auto-stamped | Codex catalog schema (per OpenAI docs) does not require a version on the plugin entry — version metadata lives in the plugin manifest at `.codex-plugin/plugin.json` which the catalog source resolves to. Adding a version field to the catalog would be redundant and would force a fourth stamping path. | Add a `plugins[].version` field and stamp it (rejected — schema-redundant). |
| Defer `verify_catalog_shape()` and CI cross-check | Primary goal (Codex install works) is met by the committed file. Defense-in-depth is nice-to-have for a hand-edited file that's not on the build path. | Adopt PR #52's `verify_catalog_shape()` skeleton with Git-URL-adapted jq expressions (deferred to follow-up issue). |
| Keep Claude marketplace at `source: "./"` | Canonical in-repo idiom; inherits `#ref` pinning from `claude plugin marketplace add fritze-dev/specshift#vX.Y.Z`; no redundant re-clone. The asymmetry between Claude (`./`) and Codex (`url`) reflects each target's idiom, not a missing-feature gap. | Convert Claude to Git URL (rejected — redundant + breaks `#ref` semantics). |
| Skip auto-PR-creation during propose | Session policy: no PR creation without explicit user request. Workflow's auto-PR step is conditional on tooling availability — treat policy gate as unavailability. | Create draft PR per WORKFLOW (rejected by policy). |
| Workspace in-tree, no worktree | System directive pins `claude/optimize-codex-marketplace-uVtJK` as the develop branch. Worktree mode would fork to a separate branch and conflict. | Create worktree per WORKFLOW (rejected by branch directive). |

## Risks & Trade-offs

- **Risk: Compile script header comments stay misleading** (lines 6–9 mention four root files; code uses three). → Mitigation: Note in CHANGELOG as a known follow-up; a small mini-patch can clean them up later without coupling to this change.
- **Risk: Spec calls out a schema (`AVAILABLE`/`ON_INSTALL`, Git-URL ending in `.git`) that the build doesn't enforce.** A maintainer could regress the catalog file silently. → Mitigation: ADR-003 Decision 6 records the rationale; a follow-up `verify_catalog_shape()` issue covers the defense-in-depth gap explicitly.
- **Risk: README's "stamps into all three root manifest/marketplace files" sentence stays as-is when readers might assume four files are stamped.** → Mitigation: The Multi-Target Distribution tree diagram (now showing four files) plus the AGENTS.md + CONSTITUTION clarification ("first three carry a `version` field stamped … the catalog has no `version` field, no automatic stamping") differentiate explicitly.
- **Risk: Issue #51 stays open with the "live smoke test" criterion unmet.** → Mitigation: That criterion is intentionally out of scope here; user has functional verification but no clean-machine smoke test in this change.
- **Risk: The catalog file's `source.url` is hard-coded to the repository's HTTPS URL.** A fork or rename would require a manual edit. → Mitigation: Acceptable — the URL is metadata, fork users edit it; ADR-003 Decision 6 notes the trade-off.

## Migration Plan

No migration needed. Existing Codex consumers who already had the broken auto-discovery flow are now able to install correctly via the catalog. Existing Claude consumers are unaffected.

For developers running `bash scripts/compile-skills.sh` locally: no change in behavior — the script ignores `.agents/plugins/marketplace.json`.
