# Technical Design: Fix Squash-Merge Commit Messages

## Context

When `specshift review` merges PRs via squash, the commit message defaults to a concatenation of all individual commits — which are all WIP pipeline artifacts. This produces noisy, unreadable entries in `git log`. The merge tool (`mcp__github__merge_pull_request`) supports explicit `commit_title` and `commit_message` parameters but the review spec doesn't instruct the agent to use them. Separately, the `WIP:` commit prefix is cryptic and carries "unfinished work" connotations for what are legitimate incremental pipeline commits.

## Architecture & Components

This change touches Markdown spec and instruction files only — no executable code:

1. **`docs/specs/review-lifecycle.md`** — "Merge Execution with Mandatory Confirmation" requirement extended with squash commit message composition (already done in specs stage).
2. **`docs/specs/artifact-pipeline.md`** — "Post-Artifact Commit" and "Post-Implementation Commit" requirements updated with new commit format (already done in specs stage).
3. **`src/templates/workflow.md`** — Review action step 8 instruction. Must describe commit message composition. Template-version bump 6 → 7.
4. **`.specshift/WORKFLOW.md`** — Project-instance mirror of review action step 8.
5. **`src/skills/specshift/SKILL.md`** — Propose step 4 commit message format.
6. **`.claude/skills/specshift/`** — Regenerated by `bash scripts/compile-skills.sh` (actions/review.md, actions/apply.md, SKILL.md).

## Goals & Success Metrics

* After implementation, `grep -r "WIP:" src/ docs/specs/` returns zero matches in spec requirement text and instruction text.
* The compiled actions at `.claude/skills/specshift/actions/review.md` contain the squash commit message composition text.
* `bash scripts/compile-skills.sh` succeeds without warnings.
* `src/templates/workflow.md` has `template-version: 7`.

## Non-Goals

- Configurable merge method in WORKFLOW.md frontmatter
- Retroactive cleanup of existing WIP commits in git history
- Draft PR body format change (currently `WIP: <change-name>`)
- Changes to `src/actions/review.md` links file (existing link already covers the extended requirement)

## Decisions

| Decision | Rationale | Alternatives |
|----------|-----------|--------------|
| Commit format `specshift(<change-name>): <artifact-id>` | Conventional-commit-compatible, machine-parseable, identifies source system, removes "unfinished work" stigma | Keep `WIP:` (rejected: cryptic); `draft:` prefix (rejected: no convention compliance); `chore():` (rejected: doesn't identify SpecShift as source) |
| Squash commit title: `<PR title> (#<PR number>)` | Matches GitHub's standard squash title convention; PR title is set during draft-to-ready from change name | Use change name directly (rejected: PR title may have been edited, PR number is useful for traceability) |
| Squash commit body: Why + What Changes + issue refs | Rich structured data already available in proposal.md at merge time | PR body only (rejected: may be stale or abbreviated); Changelog entry (rejected: may not exist yet at merge time) |
| Explicit `merge_method: squash` in spec | Removes ambiguity; the project uses squash merges consistently | Leave unspecified (rejected: was the root cause of the issue) |

## Risks & Trade-offs

- [Long commit bodies] → Mitigated: Why is 1-2 sentences, What Changes is typically 3-8 bullets. Total under 20 lines.
- [Proposal sections could be empty] → Mitigated: Spec edge case defines fallback chain (proposal → PR body → PR title only).
- [Breaking existing agent behavior] → Low risk: the agent reads the instruction at runtime; updated text is picked up immediately.
