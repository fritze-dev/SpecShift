## Why

Squash-merge commit messages contain raw WIP pipeline history (e.g., `* WIP: change-name — research`, `* WIP: change-name — proposal`) instead of a clean summary, because the review action's merge step doesn't compose a commit message. Additionally, the WIP commit format itself is cryptic and provides no useful context in PR commit lists. Closes #31.

## What Changes

- Add squash commit message composition to the "Merge Execution with Mandatory Confirmation" requirement — title from PR title + number, body from proposal Why + What Changes sections + issue references
- Explicitly prescribe `squash` as the merge method in the review spec
- Redesign pipeline commit message format from `WIP: <change-name> — <artifact-id>` to `specshift(<change-name>): <artifact-id>` (conventional-commit-style, machine-parseable, no "unfinished work" stigma)
- Update the implementation commit format from `WIP: <change-name> — implementation` to `specshift(<change-name>): implementation`
- Update review action instruction in both `src/templates/workflow.md` and `.specshift/WORKFLOW.md` to include commit message composition step
- Update SKILL.md propose step 4 to use the new commit format
- Add edge case for missing proposal sections (fallback to PR body)
