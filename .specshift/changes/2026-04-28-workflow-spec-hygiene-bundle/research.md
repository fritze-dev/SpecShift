## 2. External Research

Not applicable. This is a pure spec/template hygiene change — no external libraries, APIs, or reference implementations involved. The pattern itself (semantic vs. positional IDs) is well-established in the existing repo: specs already use `### Requirement: <Name>` / `#### Scenario: <Name>` and have not suffered rot.

## 3. Approaches

| Approach | Pro | Contra |
|----------|-----|--------|
| Drop numerical/letter prefixes everywhere; rely on heading text only | Simple; matches existing spec pattern; all rot risk eliminated | Ordering of presentation must come from doc structure alone (no longer encoded in headings) |
| Keep numerical IDs but add semantic anchors | Backwards-compatible for any external linkers | No external linkers exist; doubles maintenance burden; defeats purpose |
| Bump only the headings, leave instruction self-references on numbers | Smaller diff | Self-references silently rot (the very issue under fix) |
| Bump template-versions independently per template (two sequential changes) | Smaller diff per change | Two consumer-merge prompts; duplicate audit work; #59 + #58 share `workflow.md` so independent sequencing creates rebase pain |

**Chosen:** Approach 1 (drop prefixes) + bundle both issues so consumer sees one merge prompt and one audit pass.

For the Self-Check enforcement (#58), three sub-approaches were considered:

| Approach | Pro | Contra |
|----------|-----|--------|
| PR comment marker `<!-- specshift:self-check -->` | Anchored on HEAD commit; re-entrant friendly; analogous to existing `<!-- specshift:review-summary -->` | Requires GitHub tooling (already required for review action) |
| Local file `.specshift/changes/<slug>/self-check.md` | Works without GitHub tooling | Stale across HEAD updates; not visible to reviewers; pre-merge gate would have to inspect the change directory |
| Router-level sub-skill auto-dispatch | Fully automatic | Explicitly marked too invasive in Issue #58; touches `src/skills/specshift/SKILL.md` which CONSTITUTION marks immutable |

**Chosen:** PR comment marker (Approach 1).

## 4. Risks & Constraints

**Risks:**
- **Template-version cascade:** Bumping five templates triggers consumer merge prompts on next `specshift init`. Bundle minimizes this from "two independent passes" to "one pass". Documented in CHANGELOG.
- **Spec wording drift:** The new "Self-Check Mandatory After Comment Processing" requirement must not duplicate the existing "Review Comment Processing" requirement's self-check sentence. Mitigation: replace advisory wording in the existing requirement with a normative reference to the new one.
- **`.specshift/WORKFLOW.md` divergence:** Project mirror has a `4. Compile` step that the upstream template does not. When converting `src/templates/workflow.md` numbered list to bullets, the project mirror needs the same treatment plus retention of its extra `Compile` bullet.
- **Compile script enforcement:** `scripts/compile-skills.sh` validates that modified templates have bumped `template-version` against `main`. Forgetting any of the five bumps fails the build — caught in apply.

**Constraints:**
- Three-layer-architecture spec contains "exactly 8 artifact IDs" as a normative requirement; converting to a semantic statement requires careful wording so we don't accidentally allow more/fewer stages by spec.
- The `pipeline:` YAML array in workflow frontmatter must remain — order matters and is consumed by the router. We only change prose references, not the array.
- CHANGELOG historical references (`step 3`, `step 7`, `step 8` from past versions) stay — historical artifacts.
