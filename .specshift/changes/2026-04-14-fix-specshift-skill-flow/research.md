# Research: Fix SpecShift Skill Flow Triggering

## 1. Current State

The specshift skill's `description` field in `src/skills/specshift/SKILL.md` (line 3) controls how the skill appears in Claude's system prompt:

```
description: Central workflow command. Use with action argument: init (setup project), propose (create change + artifacts), apply (implement + verify), finalize (changelog + docs + version). Example: specshift propose
```

This is a passive description — it only tells Claude what the skill does, not when to invoke it proactively. After plan mode exits, Claude sees the implementation request and directly edits files instead of routing through the specshift skill.

**Affected files:**
- `src/skills/specshift/SKILL.md` — source skill definition (line 3, `description` field)
- `CLAUDE.md` — project instructions (line 5, workflow enforcement rule)
- `src/templates/claude.md` — consumer CLAUDE.md template (line 17, same rule)
- `.claude/skills/specshift/SKILL.md` — compiled release (regenerated by `compile-skills.sh`)

**Comparison with working skill triggers:**
The `claude-api` skill in the Claude Code system uses explicit TRIGGER/DO NOT TRIGGER conditions in its description field, which causes Claude to proactively invoke the skill when trigger conditions match — even without the user typing the slash command.

## 2. External Research

The Claude Code skill system uses the `description` field from SKILL.md frontmatter as the hint that appears in the system-reminder. This description controls:
1. Whether the skill shows up as user-invocable (always, if listed)
2. Whether Claude proactively invokes it (only if TRIGGER conditions are present)

The system-reminder format for skills is: `- <name>: <description>`. The TRIGGER/DO NOT TRIGGER pattern is a convention that Claude recognizes and acts on.

## 3. Approaches

| Approach | Pro | Contra |
|----------|-----|--------|
| A: Add TRIGGER/DO NOT TRIGGER to SKILL.md description | Directly addresses root cause; proven pattern (claude-api skill); no runtime overhead | Description becomes longer; need to get trigger conditions right to avoid false positives |
| B: Strengthen CLAUDE.md wording | Reinforces instruction layer; covers all file types explicitly | CLAUDE.md alone wasn't sufficient (current state proves this) |
| C: Add PreToolUse hook in .claude/settings.json | Technical enforcement layer; can't be ignored | Previous hook was removed (unreliable); hooks can't programmatically invoke skills; adds complexity |
| A+B combined | Two independent enforcement layers; addresses both the trigger gap and the instruction gap | Slightly more changes, but all targeted |

**Recommended: A+B combined** — Both changes are minimal, targeted, and address different failure modes.

## 4. Risks & Constraints

- **Router immutability rule** (CONSTITUTION.md): "The workflow skill (skills/specshift/SKILL.md) is generic plugin code shared across all consumers. They MUST NOT be modified for project-specific behavior." — Our change modifies the plugin-level `description` field which is generic behavior, not project-specific. This is compliant.
- **Template-version discipline**: Modifying `src/templates/claude.md` requires bumping `template-version` from 1 to 2.
- **Over-triggering risk**: If TRIGGER conditions are too broad, the skill could fire during casual code exploration. The DO NOT TRIGGER conditions mitigate this.
- **Consumer impact**: Consumer projects will get the updated CLAUDE.md template on next `specshift init` (template-version merge detection).
