# Technical Design: Fix Review Action Friction

## Context

Three sequencing bugs in the `specshift review` action affect the PR review-to-merge lifecycle. The review action is defined by a spec (`docs/specs/review-lifecycle.md`) and an instruction section in the workflow template (`src/templates/workflow.md` + `.specshift/WORKFLOW.md`). The spec has already been amended (v2 → v3). This design covers the instruction text changes.

## Architecture & Components

All changes are text amendments — no code, no new files, no structural changes.

**Files to modify:**
- `src/templates/workflow.md` — `## Action: review` instruction (lines 86-103), template-version 7 → 8
- `.specshift/WORKFLOW.md` — mirrored instruction section (lines 85-102), template-version 7 → 8

**Files regenerated by compilation:**
- `.claude/skills/specshift/actions/review.md` — auto-extracted from amended spec
- `.claude/skills/specshift/templates/workflow.md` — copied from `src/templates/workflow.md`

## Goals & Success Metrics

- Compiled action file includes both new scenarios (clean-tree, review-pending gate): PASS/FAIL by reading `.claude/skills/specshift/actions/review.md` after compilation
- Both workflow files have template-version 8: PASS/FAIL by reading frontmatter
- `scripts/compile-skills.sh` exits 0: PASS/FAIL

## Non-Goals

- Changing auto_approve behavior for propose/apply/finalize actions
- Adding new configuration fields
- Refactoring the review action structure

## Decisions

| Decision | Rationale | Alternatives |
|----------|-----------|--------------|
| Insert clean-tree check as new bullet between Draft transition and Review dispatch | Matches the logical flow — dirty tree must be resolved before requesting review | Could add to Draft transition bullet, but that conflates two concerns |
| Insert review-pending gate as new bullet between Pre-merge summary and Merge confirmation | Gate sits at the right point — after all review processing, before merge offer | Could add to Merge confirmation bullet, but dedicated bullet is clearer |
| Replace vague "no reviews pending or needed" with explicit `request_review` config reference | Uses existing config as the branch condition — no new config needed | Could add a separate `skip_review_on_auto_approve` config, but unnecessary |
| Move "Set proposal status: completed" to before merge in instruction | Ensures status change is included in squash commit, eliminating extra post-merge commit on main | Keep after merge — but this creates the friction described in issue #41 |

## Risks & Trade-offs

- Self-referential change: The review action instruction changes while this change will be reviewed using that instruction. After `apply` commits the changes to `.specshift/WORKFLOW.md`, the updated instruction takes effect for the subsequent `review` invocation. This is desirable — the fix validates itself. → No mitigation needed.
